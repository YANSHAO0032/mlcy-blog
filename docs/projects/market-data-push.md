# 行情推送系统：WebSocket、Redis 与 Kafka 实践

## Problem

行情服务需要把成交、盘口、K 线等数据低延迟推送给大量客户端。如果设计不当，很容易出现消息堆积、客户端雪崩、热点交易对拖垮整体服务等问题。

## Explanation

行情链路通常从撮合结果或行情聚合服务开始，通过 Kafka 承接数据流，再由行情计算服务生成 ticker、depth、trade、kline 等主题数据。Redis 可以缓存最新快照，WebSocket 网关负责维护连接、订阅关系和增量推送。

为了降低压力，网关层要按交易对和频道做订阅分组，相同数据只处理一次，再广播给多个连接。对于高频盘口数据，可以采用增量推送 + 定期快照校准的方式，避免每次都推完整深度。客户端过慢时要有丢弃、限速或断开策略，防止单个慢连接拖垮推送线程。

## Key Points

- Kafka 适合承接行情事件流，Redis 适合保存最新快照。
- WebSocket 网关要维护订阅关系，避免重复计算和重复序列化。
- 盘口数据适合增量推送，并配合快照校准。
- 慢客户端、热点交易对和消息堆积都需要隔离治理。

## Summary

行情推送系统的难点不只是“把消息发出去”，而是要在高频数据和大量连接下保持低延迟、可扩展和可降级。
