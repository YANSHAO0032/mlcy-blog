# Elasticsearch 检索设计

## Problem

当系统需要全文检索、复杂条件过滤、聚合统计和近实时查询时，传统数据库往往不够高效。Elasticsearch 可以解决“搜索快”的问题，但真正难的是：

- 如何设计索引和 mapping？
- 如何平衡写入延迟与查询性能？
- 如何处理分词、排序、聚合和高亮？
- 如何避免深分页、字段爆炸、集群抖动和查询慢？

## Explanation

Elasticsearch 是基于 Lucene 构建的分布式搜索引擎。它擅长全文检索、结构化过滤、聚合分析和近实时搜索，常用于商品搜索、日志检索、风控查询、链上交易查询和运营分析。

### 核心概念

| 概念 | 说明 |
| --- | --- |
| Index | 索引，类似逻辑库 |
| Document | 文档，类似一条记录 |
| Field | 字段 |
| Mapping | 字段类型定义 |
| Shard | 分片 |
| Replica | 副本 |
| Analyzer | 分词器 |

### 为什么 ES 适合搜索

ES 的核心优势来自倒排索引。数据库擅长按主键或索引精确定位，ES 更擅长从“词”反查“文档”。因此对标题、描述、标签、日志内容、交易 hash、地址、备注等字段的搜索会更高效。

### Mapping 设计

ES 中最容易出问题的地方之一就是 mapping。

常见字段类型：

- `keyword`：精确匹配、聚合、排序
- `text`：全文检索
- `long` / `integer` / `double`：数值字段
- `date`：时间字段
- `boolean`：布尔字段

例如：

```json
{
  "symbol": { "type": "keyword" },
  "title": { "type": "text" },
  "amount": { "type": "double" },
  "createdAt": { "type": "date" }
}
```

不要让动态 mapping 无限制扩张，否则容易出现字段爆炸、内存膨胀和查询异常。

### text 与 keyword

`text` 和 `keyword` 是最容易混淆的两个类型。

- `text`：会分词，适合全文搜索
- `keyword`：不分词，适合过滤、聚合、排序

例如交易对 `BTC_USDT`、订单号、用户 ID、链名称、状态值，一般应使用 `keyword`。

文章标题、描述、公告正文更适合 `text`。

### 查询方式

ES 查询通常分为两类：

- Query：参与相关性评分
- Filter：只做过滤，不算分，更适合缓存

常见实践是：

```text
能 filter 的尽量 filter
需要全文相关性的再 query
```

例如查询某用户最近 7 天的提现单：

- `userId` 用 filter
- `status` 用 filter
- `createdAt` 用 range filter

### 聚合与分析

ES 不只是搜索，还很适合聚合分析，例如：

- 每个币种的提现笔数
- 每条链的充值量
- 某段时间的风控命中统计
- 热门搜索词排行

但聚合也是消耗资源的。高基数字段聚合、复杂脚本聚合和大范围聚合都可能对集群造成明显压力。

### 深分页问题

ES 深分页很昂贵，因为它需要跳过前面大量结果。

不要直接使用很大的：

```text
from + size
```

更推荐：

- `search_after`
- scroll
- 分页游标

对于后台列表，通常应限制最大分页深度。

### 写入与刷新

ES 是近实时，不是强实时。写入后不会立刻对搜索可见，而是等 refresh 后才可查到。

因此要根据业务明确期望：

- 配置类数据是否允许秒级可见延迟？
- 交易搜索是否必须尽快可查？
- 日志检索是否接受短暂延迟？

写入性能和搜索可见性之间始终存在平衡。

### 索引分片设计

分片不是越多越好。分片太多会增加管理和查询开销，分片太少又限制并行度和容量扩展。

设计时通常考虑：

- 数据总量
- 单日增量
- 查询模式
- 生命周期管理

例如日志类数据适合按天或按月滚动索引；订单检索类数据更适合按业务索引管理。

### ES 在 Crypto 后端中的实践

在 crypto 系统中，ES 常用于：

- 订单和成交记录检索
- 充值提现记录搜索
- 链上交易 hash / 地址查询
- 风控日志和审计日志检索
- 运营搜索后台

例如用户搜索某一笔提现单，可以按：

- 用户 ID
- 币种
- 链
- 状态
- 时间范围
- 提现地址
- 交易 hash

组合查询，这类场景比传统数据库模糊搜索更适合 ES。

但要注意：ES 适合作为查询系统，不适合作为交易和资金主存储。订单、账户、流水的真相源仍然应该在数据库中，ES 更像一个面向检索和分析的投影层。

## Key Points

- Elasticsearch 的核心优势是倒排索引和近实时搜索。
- `text` 适合全文检索，`keyword` 适合过滤、聚合和排序。
- Mapping 设计不当会引发字段爆炸和性能问题。
- 深分页要谨慎，优先考虑 `search_after` 或游标方案。
- ES 适合做检索和分析，不适合做资金和订单真相源。
- 在 crypto 场景中，ES 非常适合订单、提现、链上交易和风控日志查询。

## Summary

用好 Elasticsearch，关键不在于会写 DSL，而在于围绕索引结构、字段类型、分词策略、分页方式和数据同步路径做整体设计。对于 Java 后端和 crypto 平台来说，ES 是提升检索能力和运营效率的重要中间件，但必须和数据库职责清晰分层。
