# GC 调优入门

## Problem

服务出现频繁 Full GC、接口 RT 飙高时，应该如何定位 JVM 垃圾回收问题？

## Explanation

GC 调优的第一步不是盲目改参数，而是先看对象分配模式和停顿来源。可以结合 GC 日志、堆转储和监控面板判断是新生代回收过于频繁，还是老年代对象晋升过快。常见根因包括大对象分配、缓存无法回收、线程池积压导致对象生命周期拉长等。

不同收集器的目标不同。G1 更强调可预测停顿，适合大堆低停顿场景；Parallel GC 更追求吞吐量，适合离线任务。调优时要围绕业务目标选择合适的回收器，而不是只追求“GC 次数更少”。

## Key Points

- 先看日志和监控，再决定参数调整方向。
- 分清 Minor GC、Mixed GC 和 Full GC 的成因。
- 关注对象晋升、内存泄漏和大对象分配。
- 调优目标应该和吞吐量、延迟目标绑定。

## Summary

GC 调优是数据驱动的工程问题。只有先搞清楚对象生命周期和内存压力来源，参数调整才有意义。
