# 交易平台代理增长与返佣结算系统设计

## Problem

在交易平台里，代理增长系统不是一个简单的“邀请返佣模块”。更准确地说，它是一套连接交易结果、运营规则、返佣结算和统计报表的增长基础设施，往往会同时涉及：

- 邀请关系与层级结构
- 返佣规则与结算周期
- 用户等级和运营活动
- 统计报表与后台配置
- 风控限制与防作弊

如果设计不好，很容易出现口径不一致、重复返佣、统计延迟、代理层级混乱和排障困难的问题。

## Explanation

代理增长系统本质上是“交易结果驱动的运营分账与结算系统”。它的核心输入通常不是用户主动提交的数据，而是现货交易、成交、充值、活动行为等事件。

一条更合理的链路可以是：

```text
用户邀请 / 绑定关系
-> 用户发生交易或活动行为
-> 交易事件进入增长系统
-> 返佣与等级规则计算
-> 统计结果入库
-> 后台展示 / 对账 / 发奖
```

### 1. 邀请关系建模

邀请关系是增长系统的基础。常见字段包括：

- 用户 ID
- 邀请人 ID
- 代理层级
- 绑定时间
- 渠道来源
- 关系状态

这里最关键的不是表结构，而是规则边界：

- 是否允许二次绑定
- 是否允许运营后台手动调整
- 是否支持多层代理
- 历史关系变更是否保留审计

一旦这些边界不清晰，后面的返佣和统计都会混乱。

### 2. 返佣与结算计算

返佣系统的核心难点有两个：

1. 以什么事件作为返佣触发源
2. 如何保证返佣计算幂等

在现货场景里，比较自然的触发源通常是：

- 成交事件
- 手续费结算事件
- 指定活动事件

返佣逻辑不应该直接依赖前端请求或临时查询结果，而应该依赖具有审计意义的交易结果事件。

同时必须保证：

- 同一成交事件不会重复返佣
- 补单和回放不会造成双发
- 手动修复有清晰补偿路径

### 3. 统计与报表

代理增长系统除了返佣，还要支持很多统计场景：

- 邀请用户数
- 有效交易用户数
- 手续费贡献
- 返佣总额
- 日/周/月报表
- 渠道转化率

这类数据不适合每次临时扫交易明细实时计算，通常需要：

- 明细表
- 日维度汇总表
- 用户维度汇总表
- 渠道维度汇总表

这样后台查询和运营分析会更稳定。

### 4. 配置化与后台能力

增长系统一定会不断变规则，所以后台配置能力非常重要。例如：

- 返佣比例
- 有效用户定义
- 活动时间窗口
- 层级升级条件
- 黑名单或白名单

如果这些规则全写死在代码里，运营每次改规则都要走发版，会极大拖慢业务节奏。

但配置化并不等于“随便配”。配置化一定要配合：

- 权限控制
- 操作审计
- 版本管理
- 发布生效时间

### 5. 风控与反作弊

增长系统天然容易被刷子和套利利用，所以必须考虑风控：

- 自成交刷返佣
- 批量注册刷邀请
- 多账号串联套利
- 代理层级绕过限制
- 活动规则被异常利用

这说明增长系统不是单纯的运营系统，它本质上也需要风控协同。

### 6. 和现货主链路的关系

你的实际工作背景里，这套系统是建立在现货业务之上的。这意味着最重要的一条原则是：

```text
增长系统依赖现货结果，但不要强侵入现货主链路。
```

原因很简单：

- 现货链路更关注稳定性和低延迟
- 增长链路更关注规则、统计、补偿和报表

因此两者更适合通过事件衔接，而不是放在同一个同步事务里。

## 实践建议

### 事件驱动

返佣和统计最好由标准交易事件驱动，而不是后台定时扫全表去补。这样更清晰，也更容易做实时化和幂等控制。

### 幂等设计

每一笔返佣都应该绑定唯一业务键，例如：

- 成交单号
- 手续费流水号
- 活动事件 ID

配合数据库唯一索引和幂等表，防止重复发放。

### 分层存储

至少应拆成：

- 关系层：邀请与代理关系
- 明细层：返佣、奖励、活动明细
- 汇总层：日报、月报、代理统计

这样统计效率和排障能力都会更强。

### 后台可追踪

运营后台不仅要展示数字，还要能回溯：

- 这笔返佣从哪笔成交来
- 这条代理关系什么时候绑定
- 这个等级为什么升级
- 这个活动奖励为什么发放

没有追踪能力，增长系统越大越容易失控。

## Key Points

- 代理增长系统本质上是“交易结果驱动的运营分账、返佣结算与统计系统”。
- 邀请关系、返佣、统计、活动和风控要分层建模。
- 返佣逻辑必须天然幂等，并具备审计能力。
- 配置化是必要能力，但必须配合权限和版本管理。
- 增长系统应依赖现货事件，不应强侵入现货主链路。

## Summary

代理增长系统看起来像运营模块，但真正做好它，需要同时理解交易事件、资金边界、返佣结算、统计口径、后台配置和风控约束。

对于做现货与增长方向的 Java 后端工程师来说，这类系统非常能体现真实业务能力：既要懂平台主链路，也要懂规则系统、事件驱动和运营协同。把它表述为“增长结算体系”而不是“活动模块”，会更有专业辨识度。
