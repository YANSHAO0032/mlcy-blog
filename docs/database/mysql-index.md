# MySQL 索引设计

## Problem

同一张表数据量上来之后，为什么有的查询依然很快，而有的 SQL 会突然变慢？

## Explanation

索引本质上是在存储层维护一套有序数据结构，MySQL InnoDB 默认使用 B+ 树来支持范围查询和排序。设计索引时，要让查询条件尽量命中联合索引的最左前缀，同时尽可能让查询在索引层完成，减少回表次数。

如果索引建得过多，会拖慢写入并增加维护成本；如果索引顺序不合理，则即使“建了索引”也不一定被优化器采用。所以索引设计一定要围绕高频查询路径和实际执行计划来做。

## Key Points

- 联合索引遵循最左前缀原则。
- 覆盖索引可以减少回表，提高查询效率。
- 索引不是越多越好，写入和存储成本要一并考虑。
- 使用 `EXPLAIN` 验证执行计划，避免纸面优化。

## Summary

好的索引设计是业务查询模式、数据分布和成本权衡的结果。看懂执行计划，比背索引口诀更重要。
