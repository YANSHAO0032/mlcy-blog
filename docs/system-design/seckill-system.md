# 秒杀系统设计

## Problem

在秒杀场景里，如何支撑瞬时高并发请求，同时保证库存不超卖、用户体验可接受？

## Explanation

秒杀系统的核心矛盾是流量远大于库存和下游处理能力。常见思路是先在接入层做限流和静态化，把明显无效请求挡在外面；再通过消息队列削峰，把下单动作异步化；库存校验则前移到 Redis 或内存层，快速返回抢购结果。最终订单落库时，还要依赖数据库乐观锁或唯一约束兜底，避免并发写导致超卖。

系统设计不能只盯住“高并发”三个字，还要考虑链路隔离、热点容灾和可观测性。若某个服务异常，需要快速降级，而不是把整个核心交易链路拖垮。

## Key Points

- 流量削峰依赖限流、排队和异步化。
- 库存扣减要有前置校验和数据库兜底。
- 关键链路要做隔离与降级。
- 压测和监控是上线前必须完成的工作。

## Summary

秒杀系统的关键不只是“扛住请求”，而是用分层治理把请求变得可控，再用一致性机制守住库存底线。
