# AI 工程工具链：Vibe Coding、Agent 与 Skills

## Problem

AI 编程工具已经从“代码补全”发展到“协作式工程代理”。对于 Java 后端工程师来说，真正有价值的不是让 AI 随机生成代码，而是把它纳入需求分析、方案设计、编码、测试、Review、文档沉淀和知识复用的工程流程。

很多团队使用 AI 工具效果不好，通常不是模型不够强，而是缺少边界：

- 没有把任务拆成可验证的小步骤。
- 没有让 AI 先读代码和理解上下文。
- 没有测试、Review 和回滚机制。
- 把业务决策、资金安全、权限控制完全交给 AI。
- 没有沉淀可复用的 prompts、agents 和 skills。

## Explanation

AI 工程工具链可以拆成四层：

```text
Vibe Coding -> AI Pair Programming -> Agentic Coding -> Skills / Workflow
```

`Vibe Coding` 更强调快速表达想法，让 AI 帮你把模糊需求变成可运行原型。它适合探索页面、脚本、Demo、内部工具和技术验证。

`AI Pair Programming` 更像结对开发。工程师提出目标，AI 负责补代码、解释代码、补测试、做局部重构，但最终由工程师控制设计和质量。

`Agentic Coding` 进一步让 AI 具备任务规划、代码搜索、文件编辑、测试执行和结果复盘能力。它适合处理多文件改动、文档生成、批量重构和自动化检查。

`Skills` 是把重复经验固化成可复用能力。例如“如何写一篇 Java 专题文章”、“如何审查 Spring Boot PR”、“如何排查 Redis 缓存问题”，都可以沉淀成固定流程，让 AI 每次按同样标准执行。

## Vibe Coding

Vibe Coding 的核心是快速把想法变成东西。

适合场景：

- 快速搭一个博客页面。
- 生成内部运营工具原型。
- 写一次性脚本。
- 验证一个 API 调用流程。
- 做低风险 Demo。

不适合场景：

- 资金系统核心逻辑。
- 权限、安全、风控策略。
- 数据库迁移和批量修复。
- 高并发交易链路。
- 没有测试兜底的生产改动。

Vibe Coding 的正确姿势是“快，但不盲信”。可以让 AI 快速生成第一版，但进入生产前必须补充类型检查、测试、错误处理、日志、监控和人工 Review。

## AI Pair Programming

AI Pair Programming 更适合日常开发。

常见用法：

- 让 AI 解释陌生代码。
- 根据已有代码风格补一个 service 方法。
- 生成单元测试和边界测试。
- 重构重复逻辑。
- 补充异常处理和日志。
- 写接口文档和变更说明。

比较好的提问方式：

```text
先阅读这个模块的现有代码风格，再帮我实现 X。
要求：
1. 不改变公开 API。
2. 保留现有异常语义。
3. 补充必要单元测试。
4. 最后说明改动和风险。
```

比较差的提问方式：

```text
帮我写一个交易系统。
```

AI 越强，越需要你给它清晰的边界、上下文和验收标准。

## Agentic Coding

Agentic Coding 指 AI 不只是回答问题，而是能像工程代理一样执行任务：

- 搜索代码。
- 阅读相关文件。
- 制定计划。
- 修改多个文件。
- 运行测试。
- 根据报错继续修复。
- 汇总改动和风险。

这类工作方式适合：

- 多文件重构。
- 文档站点批量调整。
- 新增一个完整专题。
- 给历史模块补测试。
- 修复构建和 CI 问题。
- 做代码库结构梳理。

但它也有风险：

- 可能误改无关文件。
- 可能生成看似合理但不符合业务规则的逻辑。
- 可能没有理解历史兼容性。
- 可能让改动范围过大，Review 成本升高。

所以 Agentic Coding 要配合：

- 干净的 git 状态。
- 明确的任务范围。
- 小步提交。
- 自动化测试。
- 人工 Review。
- 可回滚策略。

## Skills

Skills 可以理解为给 AI 的“标准操作手册”。它把重复经验沉淀成稳定流程，减少每次重新解释背景。

例如可以沉淀这些 Skills：

| Skill | 用途 |
| --- | --- |
| Java Article Skill | 按 Problem、Explanation、Key Points、Summary 写技术文章 |
| Crypto Backend Review Skill | 审查交易、账户、提现相关代码风险 |
| Spring Boot Debug Skill | 排查 Bean、配置、事务、启动失败问题 |
| Redis Design Skill | 设计缓存、限流、分布式锁和热点 Key 方案 |
| Blog Publish Skill | 修改文章、构建验证、提交前确认、发布 |

好的 Skill 应该包含：

- 什么时候使用。
- 输入需要什么。
- 操作步骤。
- 质量标准。
- 禁止事项。
- 输出格式。

Skills 的价值不只是提高速度，更重要的是稳定质量。

## AI Code Review

AI 很适合做第一轮 Review，尤其是发现：

- 空指针风险。
- 并发问题。
- 事务边界问题。
- 日志缺失。
- 异常吞掉。
- SQL 或缓存 key 设计问题。
- 测试缺口。

但 AI Review 不能替代人类 Review。涉及资金、权限、风控、隐私、安全和合规的代码，必须由熟悉业务的人最终判断。

对于 crypto 后端，AI Review 可以重点检查：

- 资金流水是否幂等。
- 状态机是否允许非法跳转。
- 提现是否存在重复广播风险。
- 链上回滚是否会重复入账。
- Redis 锁是否有 owner 校验。
- 消息重复消费是否安全。

## Prompt 工程实践

写好 prompt 的关键是给出上下文、目标和约束。

推荐结构：

```text
角色：你是资深 Java 后端工程师。
背景：这是一个 crypto 交易系统的账户模块。
目标：实现冻结余额逻辑。
约束：
- 必须幂等。
- 不能扣成负数。
- 需要记录资金流水。
- 不能改变已有接口。
验证：
- 补充单元测试。
- 说明并发风险。
```

不要只说“帮我优化一下”。优化必须明确指标：延迟、吞吐、内存、可读性、可维护性、稳定性，还是安全性。

## 与传统工具结合

AI 工具不是替代 Git、CI、测试和监控，而是和它们组合。

推荐流程：

```text
需求澄清
-> AI 辅助设计
-> 小步实现
-> 本地测试
-> AI 初审
-> 人工 Review
-> CI
-> 灰度发布
-> 监控验证
```

在这个流程里，AI 更像加速器，而不是裁判。

## Crypto 后端中的适用边界

适合 AI 加速的工作：

- 文档整理。
- 测试样例生成。
- DTO、Mapper、配置类编写。
- 低风险重构。
- 日志和异常补充。
- 内部工具和脚本。
- 代码解释和知识库沉淀。

需要谨慎的工作：

- 撮合核心逻辑。
- 资产扣减和入账。
- 提现签名和广播。
- 私钥管理。
- 风控策略。
- 权限和安全边界。
- 数据库迁移和批量修复。

这些高风险领域可以让 AI 辅助分析和生成测试，但不能让 AI 单独决定最终实现。

## Key Points

- Vibe Coding 适合快速探索，不适合直接进入生产核心链路。
- AI Pair Programming 适合日常编码、解释、测试和局部重构。
- Agentic Coding 适合多文件任务，但必须控制范围和验证结果。
- Skills 可以把个人经验沉淀成可复用流程。
- AI Review 可以提高发现问题的概率，但不能替代人工 Review。
- 在 crypto 后端中，资金、安全、风控和链上状态必须有人类兜底。

## Summary

前沿 AI 工具链的价值不是“替你写所有代码”，而是把工程师从重复劳动中解放出来，让更多精力投入架构判断、业务建模、风险控制和质量保障。

对 Java 后端工程师来说，最专业的 AI 使用方式不是炫技，而是建立一套可验证、可回滚、可审查、可沉淀的工程流程。AI 负责提速，人负责边界和责任。
