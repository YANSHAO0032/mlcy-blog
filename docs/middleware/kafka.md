# Kafka 消息队列设计

## Problem

Kafka 在后端系统里常用来解耦服务、削峰填谷、异步处理和构建事件流。但真正上线后，问题往往不是“消息能不能发出去”，而是：

- 如何保证消息不丢？
- 如何处理重复消费？
- 如何设计分区和消费组？
- 如何平衡吞吐、顺序和延迟？
- 如何处理堆积、回溯和重放？

## Explanation

Kafka 本质上是一个分布式日志系统。Producer 把消息写入 Topic，Topic 再按 Partition 分片存储，Consumer Group 以组的形式消费分区。它不是传统意义上“把消息从 A 推给 B”的黑盒，而是一套围绕顺序写磁盘、分区并行、批量网络传输构建的高吞吐事件平台。

### 核心概念

| 概念 | 说明 |
| --- | --- |
| Topic | 逻辑上的消息主题 |
| Partition | Topic 的分区，决定并行度和局部顺序 |
| Offset | 分区内消息位点 |
| Producer | 发送消息的客户端 |
| Consumer | 消费消息的客户端 |
| Consumer Group | 一组协同消费的消费者 |
| Broker | Kafka 节点 |
| Replica | 分区副本，用于高可用 |

### 为什么 Kafka 吞吐高

Kafka 高吞吐主要来自：

- 顺序写磁盘
- Page Cache
- 批量发送与批量拉取
- 零拷贝
- 分区并行

因此 Kafka 非常适合日志、埋点、交易事件、行情流、风控流水等高吞吐场景。

### 分区与顺序

Kafka 只保证单个 Partition 内的有序性，不保证 Topic 全局有序。

如果业务要求某一类消息严格顺序，例如：

- 同一订单的状态变更
- 同一账户的资金流水
- 同一提现单的审核与广播状态

就必须让这些消息路由到同一个分区。通常做法是使用业务主键作为 key，让 Kafka 按 key 分区。

### 消费语义

Kafka 常见语义：

- At most once：最多一次，可能丢消息
- At least once：至少一次，可能重复消费
- Exactly once：严格一次，语义更复杂，成本更高

工程里最常用的是：

```text
至少一次 + 消费端幂等
```

也就是说，系统接受消息重复，但不允许重复产生业务结果。

### 生产端可靠性

Producer 可靠性主要依赖：

- `acks=all`
- 合理的重试次数
- 幂等生产者 `enable.idempotence=true`
- 批量参数和超时配置

如果是交易、充值、提现等关键事件，不应为了低延迟随意降低可靠性配置。

### 消费端设计

Consumer 的关键点包括：

- 手动提交 offset 还是自动提交
- 业务处理成功后再提交 offset
- 幂等设计
- 失败重试和死信队列

自动提交 offset 容易造成“消息还没处理完，位点已经提交”。更稳妥的方式通常是处理成功后手动提交。

### 消息堆积

Kafka 堆积是最常见的线上问题之一。常见原因：

- 消费速度跟不上生产速度
- 某个分区热点严重
- 下游数据库/接口变慢
- 消费端异常重试阻塞

解决方式：

- 提高消费并行度
- 优化单条处理逻辑
- 扩容分区
- 做热点 key 打散
- 降级非核心消费链路

### 回溯与重放

Kafka 的价值之一是可以通过 offset 回放消息。这对数据修复、风控重算、历史补偿非常有用。

但回放必须小心：

- 下游必须幂等
- 要明确回放范围
- 不要和实时消费相互污染
- 最好隔离消费组

### Kafka 在 Crypto 后端中的实践

在 crypto 系统里，Kafka 很适合承接这些链路：

- 订单状态变更事件
- 成交回报与行情推送事件
- 账户流水异步通知
- 充值提现状态广播
- 风控命中事件
- 埋点和审计日志

例如撮合结果可以先写入 Kafka，再由行情服务、账户服务、通知服务分别消费，避免撮合链路被太多下游同步调用拖慢。

但要注意：Kafka 可以解耦，不等于天然一致。资金变动仍要先在本地事务中落库，再通过可靠事件表或事务消息把事件送入 Kafka。

## Key Points

- Kafka 本质是分布式日志系统，不只是传统 MQ。
- 顺序性只在单分区内成立，业务有序要靠合理分区键设计。
- 最常见的可靠性方案是“至少一次 + 消费端幂等”。
- 自动提交 offset 简单，但在关键链路上风险更高。
- 堆积、热点分区、回放污染是常见线上问题。
- 在 crypto 场景中，Kafka 非常适合作为交易、行情和风控事件总线。

## Summary

Kafka 的专业使用方式，不是会建 Topic 和发消息，而是能围绕分区、顺序、幂等、可靠性和回放能力设计完整事件链路。对于高并发 Java 后端和 crypto 平台来说，Kafka 是连接核心业务与异步生态的关键基础设施。
