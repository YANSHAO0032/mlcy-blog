# Crypto 交易系统整体架构设计

## Problem

Crypto 交易平台的后端系统既要处理高并发下单和行情推送，也要保证资产账户不出错。相比普通电商订单系统，交易系统更关注低延迟、强审计、状态一致性和资金安全。

## Explanation

一个典型交易系统可以拆成接入层、订单服务、撮合引擎、账户服务、行情服务、风控服务和清结算链路。用户下单后，订单服务先做参数校验、权限校验和幂等控制，再调用账户服务冻结资产。冻结成功后，订单进入撮合引擎，撮合结果会生成成交记录和资金变动事件，最后由账户服务完成入账、解冻和手续费扣减。

系统设计的关键不是把所有逻辑塞进一个事务，而是让每个阶段都有明确状态机、唯一幂等键和可追溯流水。撮合结果、账户变动、订单状态变更都应该形成事件流，便于异步消费、审计追踪和故障恢复。

## Key Points

- 订单、撮合、账户、行情、风控要明确边界。
- 资产冻结和成交入账必须具备幂等与审计能力。
- 撮合链路关注低延迟，资金链路关注一致性和可恢复。
- 关键状态变更要通过事件和流水持久化，避免只依赖内存状态。

## Summary

Crypto 交易系统的专业性体现在两个维度：交易链路要快，资金链路要稳。架构设计需要同时兼顾低延迟、幂等、一致性、审计和异常恢复。
