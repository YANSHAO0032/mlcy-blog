# 资产账户系统的幂等与一致性设计

## Problem

资产账户系统最怕的问题不是接口慢，而是资损。冻结、解冻、扣减、入账、手续费和资金流水只要有一步重复执行或漏执行，就可能造成用户余额异常。

## Explanation

账户模型通常会拆分为可用余额、冻结余额和总余额。用户下单时冻结可用余额，成交时扣减冻结余额并给对手方入账，撤单时解冻未成交部分。每一次余额变化都必须关联业务单号、资产类型、账户 ID、方向、金额和幂等键，并落一条不可变资金流水。

数据库层面可以通过唯一索引防止重复流水，通过乐观锁或条件更新防止余额被并发扣成负数。应用层要保证同一个业务事件重复投递时，只会产生一次资金变动。对于跨服务事件，建议使用本地消息表、事务消息或可靠事件表来保证状态和消息最终一致。

## Key Points

- 余额表负责当前状态，流水表负责审计和追溯。
- 所有资金变动都必须有业务幂等键。
- 条件更新可以防止并发扣减导致负余额。
- 账户事件要支持重复消费、乱序保护和异常补偿。

## Summary

资产账户系统的核心目标是“不多扣、不少扣、不重复、不丢失”。设计时要优先考虑幂等、审计、并发控制和异常恢复，而不是只追求接口吞吐量。
