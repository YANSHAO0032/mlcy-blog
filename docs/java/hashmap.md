# HashMap 原理

## Problem

面试里经常会问：`HashMap` 为什么查询快？扩容时为什么会影响性能？在高并发场景下直接使用会有什么风险？

## Explanation

`HashMap` 的核心结构是数组 + 链表 + 红黑树。键经过哈希计算后映射到数组桶位，桶位冲突时先挂链表，链表长度超过阈值并且容量足够时再树化为红黑树，从而避免极端碰撞导致的线性查找。

扩容发生在元素个数超过 `capacity * loadFactor` 时，默认负载因子是 `0.75`。扩容会创建更大的数组，并把原有元素重新分配到新桶位，所以这是一个成本较高的操作。如果容量预估不足，业务高峰期就容易出现频繁扩容。

## Key Points

- 查询效率通常接近 `O(1)`，但严重哈希冲突时会退化。
- JDK 8 之后用红黑树优化长链表，降低最坏情况复杂度。
- 扩容本质是一次 rehash，既消耗 CPU，也会带来瞬时延迟抖动。
- `HashMap` 线程不安全，多线程场景应考虑 `ConcurrentHashMap`。

## Summary

理解 `HashMap` 不能只停留在“数组加链表”这一层，更重要的是要掌握哈希分布、扩容成本和并发语义，这些点决定了它在真实系统中的适用边界。
