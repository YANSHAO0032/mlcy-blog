# Redis 缓存设计

## Problem

Redis 在后端系统里不只是“加缓存”这么简单。真正的问题通常是：

- 如何提升热点数据读取性能？
- 如何避免缓存穿透、击穿、雪崩？
- 如何处理缓存和数据库的一致性？
- 如何设计分布式锁、限流、排行榜和延迟任务？
- 如何避免大 key、热 key、慢查询和内存淘汰导致线上故障？

## Explanation

Redis 的本质是内存型数据结构服务器。它支持字符串、哈希、列表、集合、有序集合、位图、HyperLogLog、Stream 等结构，因此除了缓存，还适合做计数器、排行榜、分布式锁、消息流和限流器。

在业务系统里，Redis 最常见的角色有四种：

1. 作为缓存层，降低数据库读压力。
2. 作为高速状态存储，保存热点配置、会话、风控阈值等。
3. 作为分布式协调组件，承接锁、限流、幂等标记等场景。
4. 作为轻量消息或流式中间件，处理订阅、事件流、异步任务。

### 常见数据结构选型

| 场景 | 推荐结构 | 说明 |
| --- | --- | --- |
| 简单 KV 缓存 | `String` | 最常见，适合对象序列化后存储 |
| 用户配置/属性 | `Hash` | 字段更新更灵活 |
| 排行榜 | `ZSet` | 支持分数排序和范围查询 |
| 去重集合 | `Set` | 适合标签、用户集合 |
| 消息流 | `Stream` | 支持消费组 |
| 计数器 | `String` + `INCR` | 实现访问计数、限流 |
| 延迟任务 | `ZSet` | 通过 score 表示执行时间 |

### 缓存穿透、击穿、雪崩

缓存问题的本质是请求流量和后端承载能力不匹配。

缓存穿透通常由不存在的数据反复查询引起，可以通过布隆过滤器或空值缓存缓解。比如用户传入不存在的币种 ID，不应该每次都穿透到数据库。

缓存击穿常见于热点 key 过期瞬间，海量请求同时回源。常见处理方式包括：

- 互斥锁重建缓存
- 热点 key 永不过期
- 逻辑过期 + 后台异步刷新

缓存雪崩则是大量 key 同时失效，导致数据库瞬间被打穿。典型应对方式：

- 过期时间加随机扰动
- 多级缓存
- 回源限流与降级
- 对热点业务做预热和隔离

### 一致性问题

Redis 一般不适合作为强一致主存储，因此必须明确一致性策略。

最常见方案是：

```text
更新数据库 -> 删除缓存
```

这样做比“先更新缓存再更新数据库”更稳妥，因为缓存天然适合丢失和重建，不适合作为真相源。

如果对一致性要求更高，还可以考虑：

- 延迟双删
- 订阅 binlog 异步更新缓存
- 消息队列驱动缓存失效
- 版本号控制或逻辑时间戳

但要注意：缓存一致性通常是最终一致，而不是强一致。

### 分布式锁

Redis 分布式锁常见实现是：

```text
SET key value NX EX seconds
```

关键点不是“能加上锁”，而是：

- value 必须唯一，释放锁时校验 owner
- 必须设置过期时间，避免死锁
- 业务执行时间不能无限长
- 不要把 Redis 锁当成资金安全的唯一保障

对于账户扣减、提现状态推进等场景，Redis 锁只能做辅助保护，数据库条件更新和状态机才是核心兜底。

### 限流和幂等

Redis 很适合实现接口限流和幂等控制。

限流常见方式：

- 固定窗口
- 滑动窗口
- 漏桶
- 令牌桶

幂等场景中，可以用 Redis 记录请求唯一键，例如提现请求号、支付单号、消息消费 id。但 Redis 中的幂等标记不能替代数据库唯一索引，最好双重兜底。

### 热 key、大 key 与慢查询

线上 Redis 故障很多都来自热 key 和大 key。

热 key 问题表现为：

- 单个 key QPS 过高
- 某个 Redis 节点 CPU 飙升
- 网络带宽被热点流量打满

治理方法：

- 本地缓存二级缓存
- key 分片
- 预计算和静态化
- 热点隔离

大 key 问题表现为：

- 单次读取耗时高
- 删除阻塞
- 主从复制压力大

治理方法：

- 拆分 key
- 控制集合大小
- 使用异步删除 `UNLINK`
- 定期巡检和清理

### 淘汰策略与持久化

Redis 内存满了以后，行为由淘汰策略决定，例如：

- `noeviction`
- `allkeys-lru`
- `volatile-lru`
- `allkeys-lfu`

缓存型业务通常更关注淘汰策略，而状态型业务要谨慎依赖 Redis 内存回收。

持久化方面常见有：

- RDB：快照，恢复快，可能丢最近一段时间数据
- AOF：追加日志，数据更完整，但文件更大
- 混合持久化：兼顾两者优点

如果 Redis 承担的是缓存角色，持久化优先级可以低一些；如果承担状态存储或消息功能，则必须评估恢复语义。

### Crypto 后端中的实践

在 crypto 系统里，Redis 常用于：

- 币种配置、本地风控阈值、手续费参数缓存
- 行情快照与热点盘口缓存
- WebSocket 订阅关系和在线用户状态
- 提现审核幂等键、风控限流、短信验证码计数
- 延迟重试任务和轻量级事件队列

但涉及用户资产余额、充值入账、提现状态机等核心链路时，Redis 只能做加速和辅助状态，不能替代数据库和审计流水。

## Key Points

- Redis 不只是缓存，还能承担计数、排行榜、锁、限流和消息流能力。
- 穿透、击穿、雪崩的成因不同，治理方式也不同。
- 更新数据库后删除缓存，是更常见也更稳妥的一致性策略。
- Redis 锁可以辅助并发控制，但不能替代数据库条件更新和状态机。
- 热 key、大 key、慢查询和淘汰策略是线上 Redis 的常见风险点。
- 在 crypto 场景里，Redis 适合做高频状态和热点数据加速，不适合做资金真相源。

## Summary

Redis 是后端系统最重要的中间件之一，但它的价值不在于“会用 set/get”，而在于根据业务场景选择合适的数据结构、设计正确的一致性策略，并提前治理热点、内存和可靠性问题。真正用好 Redis，才能让系统既快又稳。
